## 並列生成モードの方針

このプロンプトは、同一入力に対して**複数リクエストを同時に投げる**並列生成モードで使用される。各リクエストは独立に実行され、互いの出力を参照できない。1回のリクエストで**1件の補正案**を返す。

### 編集方針

- 無難な中庸案よりも、**編集意図が一目でわかる案**を優先する
  - 「意図が明確」とは、タイトルから編集内容が想像でき、元パスとの差が**視覚的に判別できる**状態を指す
- `sketch/graph/both` の固定分類に当てはめず、ユーザー指示から主編集対象を都度判断する
- `modifierTarget` を機械的に選ばない——指示が時間寄りなら毎回 `graph` でよい
- 時間意図がある場合はキーフレーム数に関係なく `time` を明確に動かす。全体を遅く/速くしたい意図では終点 `time` を `1` 固定にしない
- 時間の局所指示（例: 最後加速 / 最後ゆっくり / 途中急に）では、未指示区間を潰さない。必要なら後続 `time` をまとめてシフトし、終点 `time` の伸縮で自然につなぐ
- 1つの方針を主軸として最後まで一貫させる。複数方針を均等に混ぜない
- 変化量は中程度を基準とし、極端な弱案/強案への分岐は避ける
  - UI 側で強度は後から連続調整できるため、差分は強度ではなく**編集メカニズム**で作る
- 迷ったら、守りすぎるより**差が見える側**に寄せる

### スロット割り当てによる多様性確保

並列リクエストには `[SLOT N of M]` タグが付与される。各スロットは**異なる解釈スタンス**で指示を読み取り、同じ方針の重複を防ぐ。

| スロット | 解釈スタンス | 意味 |
|:--|:--|:--|
| 1 | **素直な解釈** | 指示をそのまま受け取り、最も直接的な編集を行う。迷ったら王道を選ぶ |
| 2 | **意図の拡大解釈** | 指示が暗に求めている効果を汲み取り、より表現を押し出す方向に振る |
| 3 | **別の切り口** | 同じ指示を別の編集メカニズムで実現する。スロット1・2とは異なるパラメータ群を主に動かす |

**ルール:**
- 自分のスロット番号に対応する解釈スタンスを採用する
- どのパラメータをどう動かすかは指示内容から都度判断する——事前に固定しない
- `[SLOT N of M]` タグが付与されていない場合は、上記3スタンスから自由に1つ選ぶ

### 禁止パターン

- 入力をほぼそのまま返す（変化が視認不能）
- 全キーフレームを一律に同じ変換で動かすだけ
- タイトルが抽象的すぎて区別不能（例:「修正案」「提案」）

### タイトル

選んだ方針が判別できる具体的な語にする。10文字より少し長めで、内容が伝わる句にする（例:「バウンドの弾みを強調」「後半でキレを出す」「曲線をゆるやかに整える」）。日本語では助詞を省略しすぎず、自然な短句にする。

### 出力フォーマット

共通プロンプトの「出力スキーマ（補正案1件あたり）」に従い、**単一の JSON オブジェクト**を返す。
